§ 01 · SYSTEM · bus topology 01 / 08

Three primitives. Worker registers Function. Trigger fires Function.

A Worker is a process. A Function is a named async handler registered via iii.register_function. A Trigger is what fires it: default (direct trigger), http, subscribe (pubsub), stream, cron. Harness has no other primitives. The agent loop, the tools, the providers, the policy hooks, are all functions.

§ 02 · PRIMITIVES · function registry 02 / 08

iii has three primitives. Worker, function, trigger. Nothing else.

Every entry below is a function registered via iii.register_function. The agent::, tool::, provider::, policy:: prefixes are naming conventions for grep, not categories. The engine treats them identically. Calling them happens with iii.trigger(function_id, payload).

§ 03 · INGEST · turn pipeline 03 / 08

A single turn fans through five distinct iii triggers.

Caller fires agent::run_loop. The loop calls stream_assistant, which dispatches to a provider. The provider streams back an AssistantMessage. Tool calls are validated, fanned out via pubsub, dispatched, fanned out again, merged. Events stream back to the caller through agent::events.

§ 04 · CORE · loop state machine + hook merge 04 / 08

The loop is small, deterministic, and runs entirely on the bus.

run_loop walks an outer loop driven by steering / followup queues, and an inner loop that calls stream_assistant, dispatches tools, and re-enters until no more tool_calls arrive or max_turns is hit. Hook fan-out runs at four points; merge_before and merge_after compose subscriber replies into a single decision.

Hook fan-out · publish_collect (workaround for missing publish_collect primitive in iii-sdk 0.11.x)
replies = stream::list(name=agent::hook_reply, group_id=event_id)
while now < deadline · poll every 25 ms · deadline = 10 000 ms
decision = merge_before(replies) | merge_after(result, replies) | decode_transform(last reply)
DEFAULT_HOOK_TIMEOUT_MS = 10_000 (was 5_000 until v0.11.9). event_id is uuid v4 minted per fan-out. Source: workers/harness-runtime/src/register.rs:63, workers/harness-runtime/src/register.rs:228.

§ 05 · STATE · KV scopes + streams 05 / 08

All durable state lives in iii state KV. No private databases.

harness writes to one of three scopes (agent, models, llm-router when present). Sessions persist via session-tree. Two named streams carry per-session events and per-event hook replies. Every key is a string, every value is JSON.

§ 06 · TIERS · composability ladder 06 / 08

Each tier adds one worker. Harness code never changes.

§ 07 · SURFACES · external interfaces 07 / 08

Three binaries, one HTTP suite, one events stream.

Everything end-users touch maps to one of four surfaces. CLI for one-shots, TUI for interactive, harnessd for persistent deployments, HTTP for external dispatchers. Every surface is the same iii bus underneath.

§ 08 · BOOT · annotated startup 08 / 08

Starting harness from scratch. Numbers trace to the boot log.

Two processes, one bus. iii engine listens on port 49134 for workers and 3111 for HTTP, 3112 for streams. harness-cli opens a single websocket, registers ~40 functions, dispatches one trigger, prints events.

# 1. install (~90s on a fresh machine)
$ curl -fsSL https://raw.githubusercontent.com/iii-experimental/harness/main/install.sh | sh
[harness install] iii engine already installed at /Users/<you>/.local/bin/iii
[harness install] updating /Users/<you>/.harness/repo
[harness install] building harness, harness-tui, harnessd (release mode)
